§1. Context

engine epic folio-0dv は folio の文書規律を一般化した汎用文書生成エンジンを構築する (engine 設計 doc)。 その sub-epic folio-c5r は「任意プロジェクト向けの完全な文書一式」(canonical 約 17 doc-type) を被覆する roadmap である。 engine が現在生成できる 5 doc-type (SRS / ADR / research / principle / spec=rules) はいずれも要件索引型 — 要件・主張・原則を id で索引し、 EARS 章立てや RTM 三層連結で振る舞いを記述する文書型である。

gap — 記述型 doc-type が engine に欠けている。 多エージェント監査 (doctype-coverage-audit・wf_8e16a043) は、 汎用 suite を構成する約 17 doc-type のうち engine が生成できるのは 5 種のみで、 architecture-description が最も paradigm-breaking な穴 (P1) と確定した。 記述型は要件索引型を脱す — EARS も RTM も乗らず (構造を描く文書であって振る舞いを索引しない)、 floor が薄く ceiling 主導になりがちで、 機械保証の強度が落ちるのが最大の不確実点だった。

framing — architecture-description は「構造の WHAT」であって「HOW」でない。 constitution P-7 は design intent (WHAT) を構造・不変条件と既に規定する。 SRS = 振る舞いの WHAT / architecture = 構造の WHAT (両方とも design-intent の内側) であり、 実装 HOW (コード / SQL / ライブラリ / platform 固有) のみが外側 (P-3 / P-11)。 当初の「構造 HOW」というラベルは廃語で、 constitution の改訂は不要 (既に self-consistent)。

記述型 pack 自体が engine 初の領域ゆえ、 設計 grill (2026-06-26) は reader 論点でなく demo-walk-first で進めた — 抽象論点だけでは非エンジニアが全体像を判断できないため、 opus でモックアップを生成し :8729 で walk し、 反応で論点を確定してから本物の決定的 pack を構築した (見た目 design system の S1–S4 と同型)。 walk は照会リンクの着地バグを炙り出し、 案A (別文書照会) の確定・照会 graph = floor 背骨の発見・生成 doc に navigable id アンカーが無い問題 (folio-lzz) を顕在化させた。 clinic 診療予約 architecture を母体 (instance#1) に本物 pack を構築・検証し、 本 ADR はその実装 + 検証 + land (main 503ccb5) 後の forward decision を記録する。

§2. Decision

§2.1 schema (arc42 固定章立て + C4 図)

architecture-description pack は arc42 の固定章立て (課題と外部接点 / 全体戦略 / 構成部品 / 実行時の流れ / 設計決定 / 品質 / リスク / 用語の固定 8 章) と C4 の階層図 (mermaid で描く) を schema に採る。 固定章を採る load-bearing な理由は、 章の集合が floor の検査単位を与えること — 記述型でも「章の存在 + 各章の必須要素 (id 付き component / decision / quality / risk node) の存在」を機械検査でき、 EARS/RTM が乗らない記述型でも厚い floor を張る土台になる (§1 の最大不確実点への答え)。 図は ADR-0041 visual design system の render target (<pre class="mermaid">) として出し、 非エンジニア可読性 (gate I) を担保する。

§2.2 floor 三本柱 + 横展開 (記述型でも厚い floor)

architecture-description pack の floor は次の三本柱を持つ。

  • (a) 照会 graph 整合 — 全 cross-ref 辺 (決定 → ADR = rationale 照会 / 決定 → 要件 = 充足照会 / 決定 → 原則 = 終端) が suite 内の実在 node を role 通り指すことを機械検査する。 ADR-0043 の verify_cross_doc_refs + verify-graph.sh をそのまま流用する。 ★記述型の floor 背骨は EARS でなく照会 graph — cross-ref は文章でなく機械検査可能なグラフの辺であり、 これが記述型でも floor を厚くできる核心。
  • (b) navigable id アンカー emit — referenceable node (component / decision / quality / risk) に安定 navigable id を出し、 案A の照会を人間が実際に辿れるようにする (アンカー命名は照会 graph の node 識別子と一致させ verify が突合可能に)。 飛び先 SRS / ADR 側の id 欠落は横断課題ゆえ folio-lzz で別途閉じる。
  • (c) 固定章 + 必須要素存在 — arc42 固定 8 章の存在と、 各章の必須要素 (id 付き node) の存在を機械検査する (§2.1)。

加えて floor を次で補強する: 実装HOWリーク advisory 検出 (コード / platform mandate の本文混入を ADR-0043 の HOW-outside / P-11 4-enum 保守検出として横展開・core 無改変) / CJK inline 強調の空白規律 (assembler が強調タグの前後に空白を入れると助詞前で改行崩れを起こすため FAIL 検出・既知 CJK 語境界難所と同根) / fab-free (属性 pin と可視 echo pin の両方 = 属性が intact でも可視テキストだけ捏造する fail-open を塞ぐ)。

§2.3 案A — cross-document 照会 (再掲しない)

architecture-description は SRS の要件 / ADR の決定 / principle の原則へ cross-document link で照会するだけで再掲しない。 照会クリックで実際の別文書 (ADR / SRS / principle のページ) へ飛び、 architecture-description 自身は inline 要約も持たない。 これは WHAT → WHY → 構造の WHAT → 実装 HOW というはしご (SRS = 振る舞いの WHAT / ADR = WHY / architecture = 構造の WHAT / 実装 = HOW) の各段が互いを指すが再掲しない構造である。 inline 要約 (案B / 案C) は飛び先文書が既に持つ cover-summary / plain 説明と二重保持になり drift 源となるため採らない (§4)。

§2.4 横断原則 (core 無改変 / clinic 先)

  • ★core (lib/ + inject-prose.sh) 無改変 — 最も異質な記述型 pack が byte-identity で挿さる。 これは ADR-0044 が宣言した engine core 抽出完了が、 要件索引型を脱した記述型 doc-type へも及ぶことの確認であり、 rule-of-three (pack 追加は core 不変) の継続実証。 pack 追加物は contract YAML + assemble-arch.sh + rolemap + verify-arch.sh + prose manifest + test-adversarial-arch.sh のみ (= land した 6 ファイル)。
  • clinic 先 → folio-self-spec は後 — 記述型 pack 自体が新規ゆえ、 具体的で境界の明確な clinic 診療予約 architecture (instance#1) で assembler を先に証明し、 最難の meta × 記述型である folio-self-spec (folio 自身の architecture 記述) は後の self-dogfood 段階で instance#2 として回収し de-risk する (B6 で確立した self-host 型を適用・engine 全体の「具体例で証明 → self-host」型と同じ)。

§3. Consequences

  • 得るもの: 記述型 doc-type でも照会 graph + 固定章で厚い floor を張れ、 機械保証が ceiling に過度依存しない。 完全 doc-suite に構造記述文書が加わり、 案A で architecture-description が SRS / ADR / principle を再掲せず照会で繋がる (drift 源を排除)。 engine core が記述型 doc-type へも無改変で及び、 engine 汎用性の射程が広がる。
  • 払うもの: rendered navigability は別レイヤー — 照会 graph は contract レベルで機械検査され正しく PASS するが、 飛び先 SRS / ADR が navigable id を出さないと人間はリンクを辿れず案A の価値が空洞化する。 これは floor の欠陥でなく横断課題 folio-lzz で別途閉じる。 pack ごとに contract を維持する。 記述型ゆえ floor は ceiling 補完を前提とし、 architecture 専用 ceiling agent は folio-7vd で制度化する。
  • ADR-worthiness (rules §10.3): hard-to-reverse (schema 〔arc42 + C4〕・floor 三本柱・案A 照会モデルは consumer が architecture doc を生成し始めると転換が高コスト) / surprising (要件索引型を脱した初の記述型 / floor 背骨が EARS でなく照会 graph / 「構造 HOW」は廃語で構造は design-intent の WHAT) / real-trade-off (案A vs 案B/C 〔§4〕・clinic 先 vs self-spec 先・EARS 流用 vs 照会 graph floor の 3 種以上の実トレードオフ) の 3 条件を満たす。

§4. Alternatives

  • (照会挙動・案B/C) inline 要約または転写を持つ — 不採用。 飛び先文書が既に cover-summary / plain 説明を持つため inline 要約は二重保持 = drift 源になる。 照会は別文書への link に徹し (案A)、 architecture は要約を持たない。
  • (floor) EARS を記述型へ流用する — 不採用。 EARS は振る舞いトリガーの型 (いつ何をするか) であって構造記述に乗らない。 記述型の floor 背骨は照会 graph (機械検査可能な辺) + 固定章/必須要素存在に据える。
  • (母体) folio-self-spec を先に構築する — 不採用。 meta × 記述型である self-spec は最難で、 記述型 pack 自体が新規の段階で de-risk できない。 境界の明確な clinic で assembler を証明してから folio-self-spec を後の self-dogfood 段階で self-host する (B6 で確立した型を適用)。
  • (framing)「構造 HOW」というラベルを使う — 廃語。 構造は P-7 の design-intent (WHAT) であり HOW ではない。 HOW は実装 (コード / platform 固有) のみで、 それは P-3 / P-11 により design-intent 空間の外側に隔離される。
