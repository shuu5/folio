本 cluster は folio framework に関する architectural decision を記録する Architecture Decision Records (ADR) を保持する。 rules.html §3 File Naming Convention に従い、 各 ADR は ADR-NNNN-<slug>.html の命名で配置する。

§1. Purpose

decisions/ は constitution P-7 (Content domain exclusivity) における「decision = why」 領域を担う (ADR-0021 で 3-domain 化、 changes/ 廃止):

  • design intent (WHAT) = 構造・不変条件 (declarative design-intent 文書)
  • decision (本 cluster、 WHY) = frozen ADR、 architectural decisions
  • exploration = spec 反映前の探索

更新提案は専用領域 (旧 changes/) を持たず、 ADR (why) + delta marker (trace) + version control review で表現する (P-7 / §6)。 実装 (HOW) とその verification は design-intent 空間の外 (P-13)。

§5. ADR Creation Procedure

いつ起票するか (worthiness) は rules §10.3 (ADR Worthiness) に従う — decision が hard-to-reverse かつ surprising-without-context かつ real-trade-off の 3 条件すべてを満たす時のみ ADR にする (ADR-0031 §2.5)。 本 §5 は起票が決まった後の手順を規定する (P-7: worthiness 基準は rules に SSoT、 ここでは重複しない)。

constitution §6 Amendment Procedure に従う (ADR-0021 で changes/ + archive/ 廃止):

  1. 変更提案を proposal として起案 (review 用、 version control branch 上。 専用 changes/ dir は持たない)。
  2. user 承認を取得 (P-10、 constitution 改訂の場合)。
  3. ADR を Accepted status で本 cluster に作成。
  4. design-intent 文書を編集し、 SemVer bump + delta marker を inline 追加。
  5. version control の review (PR / diff) で trace を確定 (archive/ への移動は行わない — git 履歴が archive を兼ねる)。

例外: folio 初の ADR (ADR-0001) は bootstrap 段階で起票されたため proposal step を skip した折衷を採用。 後続 ADR は本手順を遵守する MUST。
