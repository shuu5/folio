§1. 目的とロードマップ上の位置づけ

folio は当初「自分自身を規律する META FRAMEWORK」として作られたが、 そこで磨いた文書規律 (dual-audience / EARS / RTM / declarative / 1 名 1 語 / HOW を外側へ / 3-domain 分離 / inter-doc 照会) は folio 固有ではなく あらゆる文書一式に効く一般原理である。 文書規律エンジン (C) はこの規律を抽出し、 任意プロジェクト向けに「完璧な文書一式」を生成する道具にする試みである。

進め方は 抽出主導 (rule-of-three): 全列挙で設計するのではなく、 folio を強い事前知識として ~3 例の実物を作り、 共通項を core として後から抽出する。 SRS-pack (epic folio-c5g) が一例目、 本 B1 の ADR-pack が二例目 (instance#2)。 止め時は「新しい doc-type が core を一切変えずに純粋な pack として挿さった」とき。 抽出は可逆 (誤って core に置いた物は pack へ降格できる)。 endgame は folio 自身をこのエンジンで dogfood すること。

§2. 論点1 — エンジンの 3 層構造

engine は次の 3 層 + 別軸の constitution に分かれる。 「機構 (どう生成・検証するか)」と「規律 (何が良い文書か)」と「中身 (各 doc-type 固有の部品)」を分離するのが要点で、 これが core / pack 境界を引く基準になる。

層役割具体 (folio の現状から)
plumbing core (機構)doc-type 非依存の生成・検証の足回り現 §12 の 4 機構: ①機械 SSoT から生成 ②決定的 assembler + opus prose ③自己完結 (single-file inline) ④2-gate (floor 機械 ∧ ceiling LLM)
普遍規律層 (著作規律)どの文書にも効く「良い文書」の規律dual-audience / declarative / 1 名 1 語 (canonical name) / HOW を外側へ / domain 分離 / inter-doc 照会
doc-type pack (中身)その文書型固有の schema + 部品 + gate 詳細SRS: EARS・RTM・37 部品 / ADR: context・options・decision・consequences・supersession
(別軸) constitution採用時のみ効く不変原則 (§5)optional-but-special

core⟷pack の境界判定 = 「その機構を新 doc-type に持ち込んだとき改変が要るか」。 改変ゼロで再利用できれば core、 doc-type 固有なら pack。 B1 の実物がこの判定を初めて検証した (§7)。

§3. 論点2 — 照会 (inter-doc cross-ref) = 抽象ロール graph

文書間・文書内の「主張 → 根拠 → … → 終端」のつながり (照会) を、 doc-type 固有語ではなく 抽象ロールで定義する。 core が次の抽象役割と普遍 edge「主張 → 根拠 → 探索 → … → 終端」を定義し、 各 pack が具体語をこの役割へ写像する。

抽象ロール意味SRS-pack の写像ADR-pack の写像
claim主張・約束されるもの要件 (FR/NFR)decision (採用した判断)
rationaleなぜそれが要るか上位ニーズ (N-x)context (背景・力学)
exploration検討した選択肢(SRS では薄い)options (選択肢 + 利点/欠点)
verification満たせたと確かめる基準受入基準 (AC)(検証手法バッジ)
principle遡らない終端原則(constitution あれば)principle (照会終端)
implementationHOW 側 (外部化)tests / コード (照会リンク)実装方式 (decision が指す)

照会終端の規約 = principle があれば principle、 無ければ top-rationale で止める。 既存の folio JSON-LD graph (relations / xref) を描画・格納の基盤に再利用し、 1 名 1 語 (P-5) を保つことで文書横断クエリが可能になる。

★B1 で実証した最難関 edge = cross-doc 照会: ADR の decision (claim) が別文書である SRS の要件 FR2 (予約受付) / FR3 (競合拒否) を正当化する edge を機械 SSoT に持たせ、 参照先 SRS に当該要件 ID が実在することを fail-closed で検証した (§7)。

§4. 論点3 — HOW を外側へ (enforcement は 3-tier)

design-intent (WHAT/WHY) と実装・検証 (HOW) を content として分離し、 HOW へは照会リンクで繋ぐ (本文に HOW を埋めない)。 これを守らせる enforcement は core 同梱の 3-tier:

  • tier 1 — SessionStart 注入: 照会 map の digest (Tier-1 小) を毎セッション文脈へ注入する (shaping)。
  • tier 2 — PreToolUse 編集 gate: 編集時に content 境界を hook で gate する (shaping)。
  • tier 3 — CI floor ∧ ceiling: ★guarantee はここだけ。 hook は vim 直編集・opt-in・コストで保証にならないため、 最終的な担保は CI gate が負う。

照会 graph が tier 1 の注入物になる。 これは folio の ADR-0007 (context injection) / ADR-0028 (2-gate enforcement) を一般化したものである。

§5. 論点4 — constitution = optional-but-special

constitution (不変の最上位原則) は 必須ではない。 採用したプロジェクトに対してのみ、 core が (a) 不変性、 (b) 改訂 gate (P-10 一般化: 改訂は ADR 経由で照会記録を残す)、 (c) 照会終端としての役割、 を保証する。 採用しないプロジェクトでは照会は top-rationale で終端する (§3)。 「必須でないが、 採用時だけ特別扱い」という非対称が要点。

§6. 論点5 — versioning = 単一版ライン (additive)

engine は 単一の版ラインで進める (folio の plugin.json 1 版)。 SRS-gen は 1.1.0、 core 抽出 + ADR-pack は後続 MINOR、 と 追記式に積む。 pack は「電池同梱」で folio と版が連動する。 MAJOR (2.0) は contract schema を破壊するときだけ。

★設計制約 (B1 から守る): core 抽出は 既存 SRS contract schema を不変に保つ (抽象ロールへの写像は pack 側に置く) ことで、 非破壊の MINOR に収める。 B1 の ADR-pack はこの制約を守って作った (SRS contract に一切触れず、 ADR から SRS を照会する片方向で実装)。

§7. 論点6 + B1 実証 — instance#2 (ADR-pack) が炙り出した core / pack 境界

B1 (folio-bwc) は clinic SRS の FR2/FR3 (二重予約防止) を正当化する clinic ADR を機械 SSoT から生成し、 同一ドメインで SRS + ADR の文書ペアを成立させた。 実物 (.claude-plugin/design-system/generator/) で得た core / pack の切り分けは次の通り。

要素SRS との関係判定
inject-prose.sh (prose 注入)無改変で共用 (data-slot-id ベースで doc-type 非依存)core 候補 (強)
esc / mark_terms / band / cover / glossary / footer の語彙idiom を共用 (関数を流用)core 候補
2-gate (floor 機械 ∧ ceiling LLM) の型同型 (verify-adr が verify-fab と同じ規律)core 候補
fabrication-free 規律 (行数=contract導出 / escape / 集合一致 / fail-closed)同型だが対象 schema が違うcore 規律 + pack 適用
contract schema (context/options/decision/...)SRS と全く別pack 固有
section emitter (options card / decision panel / cross-doc justify)SRS と別構造pack 固有
cross-doc 照会の解決 (justifies → 別 contract の req 実在)SRS には無い新軸core 機構 (照会) の pack 適用

一次証拠: prose injector が 1 byte も変えずに ADR で動いたこと、 2-gate と fabrication-free の規律がそのまま移ったことが「SRS-pack ∩ ADR-pack = core」を具体的に示す。 一方 assembler 本体は doc-type 固有の section 構造ゆえ fork が要った (= pack)。 core 抽出 (後続別 bd) は inject-prose.sh + 2-gate 型 + fabrication-free 規律 + 照会機構を core へ引き上げ、 assemble の共通 idiom (esc/mark_terms/band/cover/glossary/footer) を共有ライブラリ化する方向になる。

非目標 (rule-of-three の経済性): フル ADR generator 化・複数 ADR・独立版は作らない。 B1 は最小の extraction-probe に留める。

§8. 次の一手

  • core 抽出 (非破壊): §7 の core 候補を共有層へ引き上げる。 既存 SRS contract schema 不変 (§6 制約) を守る。
  • research / principle pack: 三例目以降の doc-type で core の安定を確かめる (止め時の判定)。
  • 照会 / HOW-outside 機構: 抽象ロール graph の格納・描画・3-tier enforcement を core 機構として実装。
  • ADR 専用 ceiling agent: persona-walk-adr / fidelity-adr の制度化 (SRS の S5.2 = persona-walk-srs / fidelity-srs と同型) を core 抽出後に検討。
  • folio self dogfood: endgame として folio 自身の architecture/ をこのエンジンで生成・検証する。
  • 正式 ADR 化: 本 design-of-record で固めた engine architecture を、 実証が十分に積まれた段階で正式 ADR (P-10 user 承認) へ昇格させる。

§9. B4 — principle / constitution pack の設計合意

B4 (folio-igv) は §3 の照会終端ロール principle を実 doc-type 化する pack を新設する。 instance#4 として folio 自身の 14 不変原則を題材に取り、 principle 固有の 3 性質 (不変・終端・改訂は ADR 経由) を engine 機構で扱えることを実証する。 設計 grill (2026-06-20) で次の 6 点を合意した。

論点決定要点
① 題材folio 14 原則を read-only fidelity targetfrozen constitution.html は置換せず別アーティファクトとして生成 (P-10 安全)。 B6 self-dogfood の「源 contract」前払い。 過渡的に手書き frozen + 生成の二重表現を許容する。
② schema 粒度中核 + 改訂来歴を構造照会{id, 見出し, 宣言文, tier} + amended_by (ADR / date / approved_by)。 versioning / amendment 手続きは opus prose (短い policy 宣言のみ構造化)。
③ constitution = optional-but-specialbaseline-diff CI gateprinciples の golden と diff し、 宣言文 / tier / 増減の変化には必ず ADR 照会 + 版 bump を強制 → silent change を機械的に不可能化。 doc_type:constitution フラグ時のみ起動 (非対称)。 B0 決定③ (guarantee = CI) ④ (P-10 一般化) の機構化。
④ 照会終端機構principle node の局所整合principle の前方照会を FAIL (終端不変条件) + inbound 参照を verify_cross_doc_refs 再利用で fail-closed + amended_by は別軸の来歴 edge。 graph 全体の終端完備は B5 へ。
⑤ scopeprinciple pack のみ build + 写像仮説を記録投機 build をせず (rule-of-three 経済性)、 folio doc-type → pack 写像の仮説だけ本 doc に残す (§9.1)。
⑥ sequencingprinciple pack 先 → bcx 汎用 lint 後principle の異質さ (前方照会なし / baseline-diff / amendment edge) が SRS・ADR・research の 3 例の前提を破る = 汎用 lint を well-informed に一般化する最良の 4 例目 (lint 自身の rule-of-three)。

B6 (folio self-dogfood) は folio の全 doc-type を生成可能にする必要がある。 どの doc-type が既存 pack を再利用し、 どれが新 pack を要するかの仮説を残す。 実証は各 doc-type を dogfood する段で行い、 既存 pack に挿さらなければ rule-of-three が新 pack の必要を炙り出す。

§10. B5 — 照会 graph 終端完備 + HOW-outside enforcement の設計合意

B5 (folio-983) は §3 の照会 graph と §4 の HOW-outside enforcement を実機構化する。 設計 grill (2026-06-21) では reader 要約でなく 実 corpus (生成 5 contract) と現行 verify / hook 機構を read-only で精査する grounding sweep を 3 本 (論点④/⑤/⑥) 走らせ、 検証済み事実に基づいて 7 論点を詰めた。 sweep は地図に無かった重大事実 — 終端の 2 系統分裂 (clinic ADR の inline principle が constitution に不在)・inbound.from の完全無検査 (phantom 照会が GREEN 通過)・HOW-leak の hard gate 強制ゼロ (唯一の機構は advisory lens)・userConfig の runtime 未読 (base_url のみ実読) — を炙り出し、 論点⑥ の推奨を案A から案C へ反転させた。

B5 論点決定要点
① graph 空間 (cross-repo 境界)repo ローカル完備 + folio principle 照会だけ registry 参照各 repo 内で graph が閉じる。 consumer → folio principle の照会のみ plugin 同梱 registry への dangling 検証 (既存 vocabulary 配布レイヤーの拡張)。 cross-repo 統合は B5 射程外。 B6 (folio が consumer 兼 engine) を最も楽にする。
② 格納スキーマcontract YAML を graph 検証 SSoTHTML data-* は描画用 projection。 照会の真の SSoT は既に contract YAML (verify_cross_doc_refs が YAML を source に HTML と SET 一致)。 別表現への転写は drift 源ゆえ避ける。 scope = 生成 doc、 folio 自身の HTML graph 化は B6。
③ ロール写像宣言pack ごと rolemap (node-type → role 表)rolemap = node intrinsic role の SSoT。 scope 内 edge role は floor で edge.role == rolemap[node.type] を pin (co-author + enforce)。 既存 corpus から draft → 明示宣言 → pin が二重担保。 SRS の exploration 不在 (B0 開問F) は scan で実証解決。 rolemap = pack / リーダー = core。 role の多重定義は path-scoped で受容 (改名は follow-up)。
④ 終端完備の検証 (主題)独立 graph script (reachability 一括)contract/ glob → edge union → principle 終端への到達可能性を 1 計算で {終端完備 / 孤立 = warn / 逆方向 = 局所} に展開。 終端は domain-local principle (ADR inline 含む) を許容 (論点① repo-local と整合・inline を graph node に昇格)。 孤立は方向ゼロでなく到達可能性で定義。 amended_by は来歴 meta-edge として除外。 既存 core 関数は不変、 traversal は独立 script。
⑤ HOW-outside content gate二段 floor + ceilingfloor-1 照会リンク存在 + floor-2 P-11 4-enum の保守的検出 (明示 primitive の有無 = 意味判定の射程外ゆえ ADR-0028 の精度懸念に当たらない。 B5 実装は soft-warn 着地・hard-block 昇格は staged rollout = ADR-0043 §2.2)。 ceiling = 概念 HOW (P-3 portability) を既存 spec-review-ssot lens 強化で advisory (新 agent 不要)。 scope = spec_path・floor は CI で担保。 staged rollout (soft-warn 実測 → 新 ADR で CI hard gate)。
⑥ tier1/2 配布plugin 固定 (新 config 変数 0)tier1 照会 digest は folio prime 追記で常時 baked + progressive disclosure、 tier2 content gate は hook 1 本追加で全 consumer 自動適用。 consumer 独自 doc-type は実需要ゼロゆえ YAGNI follow-up。 userConfig は runtime 未読 (base_url のみ実読) のため案A の「userConfig 拡張」は新機構実装を要し不採用 (地図から flip)。
⑦ CI gate の floor ∧ ceiling決定的 → floor / 意味的 → ceiling を両方にgraph floor = 到達性 / dangling / 孤立 (例外的に exhaustive)・graph ceiling = 照会 note / role 真正性 (既存 fidelity-* lens)。 HOW floor = link + 4-enum・HOW ceiling = concept HOW (spec-review-ssot lens)。 guarantee は tier3 CI のみ、 ceiling は CEILING=PENDING の advisory (ADR-0028 一般化)。

§10.1 B5 が確立した横断原則

grill は 3 つの横断原則 — 境界原則 (contract/ 内 = 厳密検証 / 外 〔folio-self の inbound.from・amended_by.adr〕 = external-ref として warn・解決は B6)、 決定的 → floor (CI hard = guarantee) / 意味的 → ceiling (advisory) (graph 構造は有限ゆえ floor が例外的に exhaustive)、 core は additive 拡張のみ (既存 core 関数は byte-identity 非回帰・graph traversal は独立 script) — を確立した。 これらの確定形は ADR-0043 §2.4 が正本 (路線「い」= 実証後 ADR 昇格が完了)。 本 §10 は確定原則を二重保持せず、 grounding sweep が配布推奨を案A → 案C へ反転させた discovery の経緯記録に focus する。

§10.2 follow-up と B6 前払い

grill で炙り出した B5 scope 外の改善候補 (実装時に bd 起票判断): 抽象 role の改名 (P-5 hygiene・role の 3 重定義)、 ADR-less 孤立 SRS の warn → block 昇格 policy、 inbound.from / amended_by.adr の実在・reverse 検証 (B6 で folio spec graph 接続時)、 generated contract の HOW-leak lens (generator ceiling 側)、 ceiling lens の CI 非ゲート advisory 化、 userConfig 4 キーの runtime 未読 gap、 review_model の機械 override 欠落。 B6 前払い: 照会 graph helper は doc-type 非依存で core に置き folio 自身の横断 graph へ転用できるよう設計し、 folio-self-spec の HOW 参照 (§9 Bindings 等) の <code> 言及 → <a href> 昇格は B6 self-dogfood の deliverable とする。 本節も探索領域 (research/) の design-of-record (路線「い」) であり、 B5 の確定設計は ADR-0043 (P-10 user 承認・2026-06-22 accepted) へ昇格済。
