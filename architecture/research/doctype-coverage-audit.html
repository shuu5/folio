v1.0.0 · status: active · 北極星 folio-7wn の完全 doc-suite 被覆 roadmap の SSoT (探索領域・後段で正式化)

§0. 要旨

本書は 2026-06-24 に実施した多エージェント監査 (doctype-completeness-audit / wf_8e16a043) の design-of-record である。問い = 「folio 文書規律エンジン (C) は『任意プロジェクト向けの完全な文書一式』を生成する能力の全体像で、本当に self-spec (architecture description) 『だけ』が欠けているのか」。

★中心的発見: folio engine は完全 project doc-suite を構成する canonical 約 17 の文書型のうち 5 種しか生成できない (SRS / ADR / research / principle / spec=rules)。self-spec (architecture description) は最初かつ最も paradigm-breaking な穴だが唯一ではない。本書はその全 gap を優先度付きで SSoT 化し、埋める roadmap を定める。実装追跡は bd epic folio-c5r (post-B6 完全 doc-suite 被覆 roadmap)。

位置づけ (P-7 探索領域): 本書は探索領域 (research/) の記録であり、scope を silently に拡大するものではない。確定設計は段階ごとに grill → 正式 ADR (P-10 user 承認) へ昇格させる。コード無しプロジェクトへの射程拡大は folio-dxq で別途明文化する。

§1. canonical doc-type カタログ × folio 被覆

完全な project documentation suite を構成する canonical 文書型を業界標準 (ISO/IEC/IEEE 42010 / 29148 / 29119, arc42, C4, Diátaxis, ADR, RFC, IEEE 1016 等) に接地して体系化し、folio の現 pack 被覆と厳密照合した。structure_kind = requirement-indexed (離散正規単位を索引) / descriptive (prose+図+表 主体) / hybrid。被覆 = covered (生成可) / partial (machinery のみ) / gap (pack 無)。

§2. gap_set — 穴の二分と優先度

gap を質で二分すると engine の真のフロンティアが見える:

  • (a) 構造的に novel (要件索引型を脱す・RTM 非適用・floor が薄く ceiling 主導) = architecture-description (P1) / vision / roadmap / operations(手順型)。engine がまだ扱ったことのない doc category。
  • (b) indexed で安価 (既存 machinery 近接) = glossary (P1・machinery 既存=最安 keystone) / test-cases / risk / changelog / V&V。

優先度: P1 = architecture-description, glossary。P2 = test-cases, vision, interface, data-model。P3 = risk, changelog, operations, roadmap, standalone V&V。

(C) への直接 answer: 「self-spec だけが穴」は folio-self の狭い範囲でのみ真 (verification/relations は spec-pack fork で被覆済・self-spec のみ spec-row 0=構造的 novel)。汎用 suite では false (architecture に加え glossary・test-cases・vision 他が欠ける)。self-spec は最も architecturally significant だが、glossary (suite keystone・安価) と test-cases (SRS↔test ループ) を見落とすのは誤り。engine 設計 doc §9.1 は self-spec を「SRS 族 (claim 群)」と仮説していたが、tr0 grounding で spec-row 0 / EARS 0 = descriptive と実証され仮説は覆った。

§3. roadmap (post-B6・bd epic folio-c5r)

順序の核心 = architecture (最難) を最後に、安い keystone (glossary) を先に。

  1. 段階1 (folio-8f0・本書): gap 全体像の SSoT 化。
  2. 段階2: glossary pack (folio-1uc・最安 keystone・suite walk の cross-doc 整合 SSoT) + test-cases pack (folio-uvt) + suite demo 設計 grill (folio-yzv・clinic 母体)。indexed-heavy 一式 (SRS+ADR+research+glossary+test-cases) で suite demo を architecture 前に先行実証できる。
  3. 段階3: architecture-description 汎用 pack (folio-5uu・最難・grill + grounding-sweep 必須)。folio-self-spec を instance#1 として B6 self-dogfood と同一成果物で回収。
  4. 段階4: clinic suite 統合 walk (folio-rqj・cross-doc graph 双方向 fail-closed + 非エンジニア persona walk + fidelity ceiling 横断)。単一文書 walk が検査しない suite 規模の一貫性を検査する。

suite demo (段階2/4) の構成 = glossary / vision / architecture / SRS / ADR / research / test-cases / README。constitution は scale 依存 (clinic 単一チームでは optional)。

§4. コード無しプロジェクトへの射程 (folio-dxq へ)

核心命題 (verified): folio の文書規律は設計上すでにコード非依存。core の照会 graph が使う抽象ロール集合 GRAPH_ROLE_ALLOWLIST は claim|rationale|exploration|principle|verification|implementation で「コード」概念を含まない。「実装=コード」は SRS-pack 固有の写像にすぎず、ADR-pack は implementation を「実装方式 (非コード)」へ既に写像済。

規律ごとの translate 度: dual-audience=high (混在読者=専門家 vs 市民/議員/監査人 はコード無しドメインで常態・価値増)、canonical-name=high、declarative=high (法令/標準/憲章は本来 declarative)、P-7 3-domain=high、RTM=high (政策目標→施策→根拠→評価指標)、EARS=medium-high (規範型で high・記述型で low)、HOW-outside=high (再定義要: 「実装」が施行/運用/建造になる)。

橋頭堡 = principle/constitution pack (要件索引型を脱した唯一の実証済 pack・公共政策の基本方針/ガバナンス憲章は principle 型・既に gate I 達成)。第二 = architecture-description pack。最大の不確実点 = 純記述型 doc-type (EARS が乗らない) の floor をどう厚くするか (ceiling 依存で機械保証強度が落ちる)。詳細は folio-dxq の research note に展開する。
