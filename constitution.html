folio constitution

v0.1.0-draft · status: draft · 12 immutable principles, 3-tier boundary declarations

本文書は IETF RFC 2119 および RFC 8174 / BCP 14 に定義される MUST / SHOULD / MAY / SHALL のキーワード解釈規約に従う。本文中の大文字キーワードは規範的要件である。

§0. Reader's Guide — Stakeholder Perspectives (ISO/IEC/IEEE 42010:2022)

folio は 3 種類の reader を第一級 stakeholder として明示する。 各 spec file の <meta name="folio-stakeholders"> 属性で対象 persona を宣言する。

あなたが Developer であれば: §2 不変原則 P-1〜P-12 を読み、 plan / implementation 前に矛盾しないか確認する。 §3 SemVer Policy に従って改訂を提案する。 ADR は core/decisions/ に追加する。

あなたが AI Agent (Claude Code / 他 LLM) であれば: spec edit 前に必ず本 constitution を読む (P-11 違反検出のため)。 FOLIO_ARCHITECT_CONTEXT=folio-architect を set してから core/spec/ や projects/*/architecture/spec/ 配下を編集する (architecture-rules.html §9)。 cwd が ~/projects/local-projects/folio/ 配下であることを最初に確認する (LLM 混同回避)。

あなたが External Reviewer (security audit / legal review 等) であれば: §2 P-1 (未来 anchor) と §2 P-3 (WHAT-only) を確認した上で、 「現在挙動 SSoT」 と誤読しないこと。 spec と実装の divergence は本 framework では 想定内 である (詳細は P-1 commentary)。

§1. Purpose — なぜ folio が存在するか

folio は AI Agent 時代における architecture specification の framework である。 大規模プロジェクトでユーザーが当初要望を忘れて場当たり的修正で破綻するのを防ぐため、 「常に参照され続ける自然言語 anchor」 として機能する。

folio の中核思想は 未来理想 anchor (future-ideal anchor) である:

  • spec は実装の mirror ではなく、 設計の 北極星 (north star)。
  • spec → code 方向の drift (実装が追いついていない) は 想定内。
  • code → spec 方向の drift (実装制約が想定を変える) は user dialog で spec を更新。
  • 場当たり的逸脱 (spec 不在の改造) を防ぐのが folio の存在意義。

この思想は OpenSpec (Fission-AI) の「現在挙動 SSoT」 思想および BDD (Behavior-Driven Development) の「テスト挙動 SSoT」 思想と明確に分岐する。 採用フレームワークの選定で誤解しないこと。

§2. Twelve Immutable Principles (3-tier Boundary Declarations)

folio は 12 の不変原則を 3 tier に分類する。 tier は LLM が遵守度を判定する際の anchor となる。

§2.1 Always Tier (P-1 〜 P-7) — 不変、 例外なし

P-1: spec は未来理想の anchor である
spec/ 配下の全文書は「あるべき設計の最終形」 を declarative に記述する。 現在実装の mirror ではない。 drift があるのは当然 であり、 機械 diff は不可能かつ不要である。 検証は AI dialog accountability (P-8 で詳述) によって行う。
P-2: format は HTML + common.css のみ
spec / ADR / constitution / architecture-rules を含む全文書は HTML で記述する。 Markdown 例外: (a) FOLIO.md root marker (LLM が repo root を即特定するための plaintext)、 (b) stacks/*/vocabulary.yaml / folio.config.yaml (Zod 検証対象の machine-readable config)。 採用理由: 規模を絞れば token cost は問題にならず、 hyperlink + 図表現の豊かさが LLM/人間双方の理解を深める (cf. Anthropic 最近の声明 "specs designed for humans can be HTML")。
P-3: WHAT-only (HOW を spec に書かない)
spec/ 配下に platform 固有の HOW (Claude Code primitives / tmux / specific path / specific CLI invocation) を書かない。 判定 test: 「Claude Code を別 AI platform に移行したら spec を書き直す必要があるか? Yes なら HOW 漏れ」。 HOW は core/bindings/<platform>/ または stacks/*/patterns/ に分離する。
P-4: Declarative form (現在形宣言文のみ)
spec 本文は現在形の declarative narrative で記述する。 過去 narration (「Wave N で〜した」)、 未来 narration (「将来〜する予定」)、 wave-specific note は spec/ から禁止。 これらは changes/ proposal、 decisions/ ADR (frozen)、 または research/ に分離する。
P-5: 1 entity = 1 canonical name
同一概念に複数の名称を使用しない。 vocabulary は stacks/*/vocabulary.yaml で canonical name + forbidden synonyms を宣言する。 spec 本文で forbidden synonym が出現した場合、 specialist agent (spec-review-vocabulary) が違反として検出する。
P-6: Link integrity (orphan 禁止)
spec/ 配下の全 file は inbound link を 1 以上持たねばならない (例外: 各 cluster の README.html = entry point)。 broken link 数、 orphan 数、 trace_extra 数は CI gate で 0 を強制する。 orphan = 知識の消失と等価である。
P-7: Content domain exclusivity
spec / ADR / changes / research の 4 領域は内容を排他的に保持する: spec/ = what (WHAT-only 不変条件・構造) / decisions/ = why (frozen ADR、 過去判断) / changes/ = proposal (進行中の spec 更新提案、 dialog 凝固) / research/ = exploration (spec 反映前の探索・思考保全)。 領域の混在は spec 純度を破壊する。

§2.2 Ask-First Tier (P-8 〜 P-10) — user 承認後に変更可能

P-8: AI dialog accountability
AI Agent が spec を読んで実装案・refactor 案を提示する際、 「この案は constitution P-X と矛盾する」 「spec § Y との divergence がある」 を natural language reasoning で user に提示する責務を負う。 機械 diff の代わりに、 AI の自然言語推論が drift 検出 mechanism として機能する。
P-9: Stack overlay 追加・変更前に user 確認
stacks/*/ 配下に新 stack を追加する、 既存 stack の vocabulary / rules-overlay を変更する場合、 user に変更内容を提示して承認を得る。 stack 変更は project 横断的影響を持つため。
P-10: Constitution 自身への変更前に user 確認
本 constitution.html への変更 (12 原則の追加・削除・文言変更) は user 承認 MUST。 MAJOR / MINOR / PATCH 判定は §3 SemVer Policy に従う。 ADR が constitution を Supersede する場合も同様。

§2.3 Never Tier (P-11 〜 P-12) — 絶対禁止、 例外なし

P-11: spec / constitution に HOW を書かない (P-3 強化)
P-3 の WHAT-only 原則を絶対禁止 tier に escalate した版。 Claude Code 固有 primitives (Skill / Hook / cld-spawn)、 specific binary path、 tmux command、 OS-specific command を spec / constitution / architecture-rules に直接記述しない。 必要なら core/bindings/claude-code/ 等の platform-specific overlay に隔離する。 platform 中立性は folio の universal 化の前提である。
P-12: stacks/twill-legacy/ を active modification の対象にしない
twill plugin (~/projects/local-projects/twill/) は完全 freeze 済 historical artifact である。 折り返し移植する場合は stacks/twill-legacy/ 配下に read-only reference として格納し、 active edit の対象にしてはならない。 LLM 混同回避の structural guard。

§3. Versioning Policy (SemVer)

本 constitution.html は Semantic Versioning 2.0.0 に準拠する。

bump条件例
MAJOR原則の追加 / 削除 / tier 変更P-13 追加、 P-8 を Always tier へ昇格
MINOR原則の文言改訂 (semantic 影響あり)P-3 の判定 test 強化
PATCH誤記修正、 typo、 cross-ref 補正citation URL 更新

変更時は ADR (core/decisions/ADR-NNNN-constitution-change-*.html) を起票し、 本 constitution 内に <ins class="delta" data-delta-id="..."> で trace を inline 保存する (architecture-rules.html §5)。

§4. Amendment Procedure

  1. 変更提案を changes/<NNN>-constitution-amendment/proposal.html として起案 (proposal-only MUST、 design.html / tasks.html は optional)。
  2. user 承認を AskUserQuestion 等で取得 (P-10)。
  3. 承認後、 ADR (core/decisions/ADR-NNNN-...) を Accepted status で作成。
  4. constitution.html を編集し、 SemVer bump (§3) + delta marker (<ins class="delta">) を inline 追加。
  5. changes/ package を archive/changes/YYYY-MM-DD-NNN-... へ移動。

§5. Citations & Influences

  • RFC 2119 / RFC 8174 — MUST/SHOULD/MAY キーワード解釈 (本 constitution 全 normative section が準拠)。
  • arc42 (arc42.org) — architecture documentation 12 sections。 folio は essential 5 sections (§1/§3/§5/§9/§10) の概念を採用、 完全 12 section 採用はしない (over-spec のため)。
  • Diátaxis (diataxis.fr) — 4 documentation modes (Tutorial / Reference / How-to / Explanation)。 folio は projects/ 配下で 4 modes を物理 dir 分離する (architecture-rules.html §2)。
  • ISO/IEC/IEEE 42010:2022 — Architecture Description 国際標準、 Stakeholder Perspectives 概念を §0 で採用。
  • EARS (Easy Approach to Requirements Syntax) — Mavin らによる acceptance criteria 形式言語。 folio は core 要件記述に <aside class="ears"> 形式で採用 (architecture-rules.html §6)。
  • OpenSpec (Fission-AI) — changes/ lifecycle (proposal/design/tasks) を参考にしたが、 案 B (proposal-only MUST) に簡素化。 SSoT 解釈は OpenSpec の「現在挙動」 と分岐し、 folio は「未来理想」 を採用 (P-1)。
  • AWS Kiro (kiro.dev) — steering/ 規約 (product/tech/structure 3 文書) の発想を採用。 EARS notation も Kiro が啓発元。
  • GitHub Spec Kit — constitution.md パターンを採用 (folio は constitution.html として html-native 化)。
  • Anthropic 最近の声明 (2026) — 「人間も読む仕様書は Markdown ではなく HTML でも良い」 という方向性が folio の HTML-only 採用 (P-2) の追加根拠。
  • twill plugin (~/projects/local-projects/twill/) — folio の概念的ルーツ。 universal-ready な R-rule 群 (declarative narrative / link integrity / caller marker idea / drift 防止 v3) を概念継承するが、 twill 固有実装は採用しない (P-12 で隔離)。
