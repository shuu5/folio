folio constitution

v0.4.0-draft · status: draft · 12 immutable principles, 3-tier boundary declarations, plugin-first Layer 0

本文書は IETF RFC 2119 および RFC 8174 / BCP 14 に定義される MUST / SHOULD / MAY / SHALL のキーワード解釈規約に従う。 本文中の大文字キーワードは規範的要件である。

§0. Reader's Guide — Stakeholder Perspectives (ISO/IEC/IEEE 42010:2022)

folio は 3 種類の reader を第一級 stakeholder として明示する。 各 spec file の <meta name="folio-stakeholders"> 属性で対象 persona を宣言する。 具体的な caller marker / cwd 確認 mechanism は architecture-rules §7.3 参照 (P-11 により本書には HOW を記載しない)。

あなたが Developer であれば: §2 不変原則 P-1〜P-12 を読み、 plan / implementation 前に矛盾しないか確認する。 §5 SemVer Policy に従って改訂を提案する。 ADR を起票して変更履歴を残す。

あなたが AI Agent (Claude Code / 他 LLM) であれば: spec edit 前に本 constitution + architecture-rules を必読 (P-11 違反検出のため)。 caller context を architecture-rules §7.3 の定める方法で宣言してから spec 編集を行う。 cwd が folio repo (Layer 0) 配下か consumer project (Layer 1) 配下かを最初に確認する (LLM 混同回避)。

あなたが External Reviewer (security audit / legal review 等) であれば: §1 Purpose + P-1 (未来理想 anchor) を確認した上で、 「現在挙動 SSoT」 と誤読しないこと。 spec と実装の divergence は本 framework では 想定内 である (詳細は §1 commentary)。

§1. Purpose — なぜ folio が存在するか

folio は AI Agent 時代における architecture specification の framework である。 大規模プロジェクトでユーザーが当初要望を忘れて場当たり的修正で破綻するのを防ぐため、 「常に参照され続ける自然言語 anchor」 として機能する。

folio の中核思想は 未来理想 anchor (future-ideal anchor) である:

  • spec は実装の mirror ではなく、 設計の 北極星 (north star)。
  • spec → code 方向の drift (実装が追いついていない) は 想定内。
  • code → spec 方向の drift (実装制約が想定を変える) は user dialog で spec を更新。
  • 場当たり的逸脱 (spec 不在の改造) を防ぐのが folio の存在意義。

§2. Twelve Immutable Principles (3-tier Boundary Declarations)

folio は 12 の不変原則を 3 tier に分類する。 tier は LLM および人間が遵守度を判定する際の anchor となる。

§2.1 Always Tier (P-1〜P-7) — 不変、 例外なし

P-1: spec は未来理想の anchor である
spec/ 配下の全文書は「あるべき設計の最終形」 を declarative に記述する。 現在実装の mirror ではない。 drift は当然であり、 機械 diff は不要。 検証は AI dialog accountability (P-8) によって行う。
P-2: format は HTML + common.css のみ
spec / ADR / constitution / architecture-rules / changes proposal を含む全文書は HTML で記述する。 Markdown 例外: (a) FOLIO.md root marker、 (b) README.md (GitHub 表示用)、 (c) CLAUDE.md (Claude Code project instructions、 P-11 root 例外)、 (d) folio.config.yaml 等の YAML config。 採用理由: 規模を絞れば token cost は問題にならず、 hyperlink + 図表現の豊かさが LLM/人間双方の理解を深める。
P-3: WHAT-only (HOW を spec に書かない)
spec / constitution / architecture-rules に platform 固有の HOW を書かない。 判定 test: 「同等の framework を別 AI platform に移行したら spec を書き直す必要があるか? Yes なら HOW 漏れ」。 HOW (env var 名、 具体 binary path、 specific CLI invocation 等) は .claude-plugin/ harness または CLAUDE.md binding に分離する。
P-4: Declarative form (現在形宣言文のみ)
spec 本文は現在形の declarative narrative で記述する。 過去 narration (「Wave N で〜した」)、 未来 narration (「将来〜する予定」)、 wave-specific note は spec/ から禁止。 これらは changes/ proposal、 decisions/ ADR (frozen)、 または research/ に分離する。
P-5: 1 entity = 1 canonical name
同一概念に複数の名称を使用しない。 canonical name + forbidden synonyms の宣言は consumer project の vocabulary file で管理する。 spec 本文で forbidden synonym が出現した場合、 vocabulary review が違反として検出する (詳細 mechanism は architecture-rules §7.2)。
P-6: Link integrity (orphan 禁止)
spec/ 配下の全 file は inbound link を 1 以上持たねばならない (例外: 各 cluster の README.html = entry point)。 broken link 数、 orphan 数、 trace_extra 数は CI gate で 0 を強制する (architecture-rules §7.5)。 orphan = 知識の消失と等価である。
P-7: Content domain exclusivity
spec / ADR / changes / research の 4 領域は内容を排他的に保持する: spec/ = what (WHAT-only 不変条件・構造) / decisions/ = why (frozen ADR、 過去判断) / changes/ = proposal (進行中の spec 更新提案、 dialog 凝固) / research/ = exploration (spec 反映前の探索・思考保全)。 領域の混在は spec 純度を破壊する。

§2.2 Ask-First Tier (P-8〜P-10) — user 承認後に変更可能

P-8: AI dialog accountability
AI Agent が spec を読んで実装案・refactor 案を提示する際、 「この案は constitution P-X と矛盾する」 「spec § Y との divergence がある」 を natural language reasoning で user に提示する責務を負う。 機械 diff の代わりに、 AI の自然言語推論が drift 検出 mechanism として機能する。
P-9: Layer 1 consumer 固有 config 変更前に user 確認
各 Layer 1 consumer project の folio.config.yaml や project 固有 vocabulary file の変更時、 user に変更内容を提示して承認を得る。 config 変更は project-wide 影響を持つため。
P-10: Constitution 自身への変更前に user 確認
本 constitution.html への変更 (12 原則の追加・削除・文言変更・tier 変更) は user 承認 MUST。 MAJOR / MINOR / PATCH 判定は §5 SemVer Policy に従う。 ADR が constitution を Supersede する場合も同様。

§2.3 Never Tier (P-11〜P-12) — 絶対禁止、 例外なし

P-11: spec / constitution / architecture-rules に HOW を書かない (P-3 強化)
P-3 の WHAT-only 原則を絶対禁止 tier に escalate した版。 platform 固有 primitives (specific tool 名、 binary path、 OS-specific command、 env var 名等の具体値) を spec / constitution / architecture-rules の本文に直接記述しない。 必要なら .claude-plugin/ harness または CLAUDE.md binding (P-2 例外) に隔離する。 platform 中立性は folio の universal 化の前提である。
P-12: Layer 0/1 boundary 侵犯不可 + folio は rules + plugin harness を一体提供する
folio Layer 0 は rules (constitution + architecture-rules) と plugin harness (.claude-plugin/) を一体配布する (ESLint / Stylelint pattern)。 Layer 1 consumer project は別 git repo として展開され、 folio repo 内に同居してはならない。 また、 旧 twill plugin (~/projects/local-projects/twill/) は完全 freeze 済 historical artifact であり、 folio から concept 継承する場合も active edit せず reference のみとする (LLM 混同回避の structural guard、 architecture-rules §7.4)。

§3. Layer Architecture (2-Layer 体系)

folio は 2 層構造 を採用する: Layer 0 = folio (META FRAMEWORK) および Layer 1 = consumer projects (各別 repo)。 Layer 0.5 等の小数点表記は採用しない (粒度表現として不適切、 user feedback で確定)。 harness は Layer 0 の構成要素として P-12 で一体配布される。

各 layer の役割・directory layout・consumer project の構造詳細は architecture-rules.html §1 および §2 で規定する。

§4. Folio as a Plugin (P-12 の構造表現)

folio は Claude Code plugin として配布される (P-12)。 ESLint や Stylelint と同じ pattern で、 「rules + harness 一体配布」 を実現する。 consumer project は folio.config.yaml 1 file の宣言で folio plugin を採用し、 spec 編集 + harness enforce の両方を得る。

MCP (Model Context Protocol) は採用しない。 network API 制約を本 framework は許容せず、 harness は skill + hook + CLI (local process) のみで構成される。 同等の framework を別 AI platform (Cursor / Continue 等) へ移植する場合は、 platform-specific binding layer (将来の bindings/) を追加することで実現する。

§5. Versioning Policy (SemVer)

本 constitution.html は Semantic Versioning 2.0.0 に準拠する。

bump条件例
MAJOR原則の追加 / 削除 / tier 変更P-13 追加、 P-8 を Always tier へ昇格
MINOR原則の文言改訂 (semantic 影響あり)P-3 の判定 test 強化
PATCH誤記修正、 typo、 cross-ref 補正citation URL 更新

1.0.0 未満は development phase として MAJOR/MINOR/PATCH の boundary を緩める (BC break を MINOR で許容)。 1.0.0 以降は strict SemVer。

変更時は ADR を起票し、 本 constitution 内に delta marker を inline 保存する (architecture-rules §5 Delta Marker Markup)。

§6. Amendment Procedure

  1. 変更提案を changes/ 配下に proposal として起案 (proposal-only MUST、 design + tasks は optional)。
  2. user 承認を取得 (P-10)。
  3. 承認後、 ADR を Accepted status で作成。
  4. constitution.html を編集し、 SemVer bump (§5) + delta marker を inline 追加。
  5. changes/ package を archive/ へ移動。

各 step の具体的 mechanism (delta marker syntax、 caller marker 適用方法、 specialist review 起動) は architecture-rules §7.1 7-phase PR Cycle および関連 section に規定する。

§7. Citations & Influences

  • RFC 2119 / RFC 8174 — MUST/SHOULD/MAY キーワード解釈 (本 constitution 全 normative section が準拠)。
  • arc42 (arc42.org) — architecture documentation 12 sections。 folio は essential 5 sections (§1/§3/§5/§9/§10) の概念を採用、 完全 12 section 採用はしない (over-spec のため)。
  • Diátaxis (diataxis.fr) — 4 documentation modes (Tutorial / Reference / How-to / Explanation)。 各 Layer 1 consumer project で物理 dir 分離する (architecture-rules §2)。
  • ISO/IEC/IEEE 42010:2022 — Architecture Description 国際標準、 Stakeholder Perspectives 概念を §0 で採用。
  • EARS (Easy Approach to Requirements Syntax) — Mavin らによる acceptance criteria 形式言語 (architecture-rules §6)。
  • OpenSpec (Fission-AI) — changes/ lifecycle (proposal/design/tasks) を参考にしたが、 proposal-only MUST に簡素化。 SSoT 解釈は OpenSpec の「現在挙動」 と分岐し、 folio は「未来理想」 を採用 (P-1)。
  • AWS Kiro (kiro.dev) — steering/ 規約 (product/tech/structure 3 文書) の発想を採用 (各 consumer project)。 EARS notation も Kiro 啓発元。
  • GitHub Spec Kit — constitution.md パターンを採用 (folio は constitution.html として html-native 化)。
  • ESLint / Stylelint — rules + plugin engine 一体配布 pattern (P-12)。 folio plugin 構造の直接 reference。
  • Anthropic 公式声明 (2026) — 「人間も読む仕様書は Markdown ではなく HTML でも良い」 という方向性が folio の HTML-only 採用 (P-2) の追加根拠。
  • twill plugin — folio の概念的ルーツ (P-12 で read-only reference として参照)。 universal-ready な原則 (declarative narrative、 link integrity、 caller marker idea、 drift 防止) を概念継承するが、 twill 固有実装は採用しない。
