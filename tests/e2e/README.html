tests/e2e/ — agent-driven e2e integration test

v0.1.0-draft · status: active · REQ-VER-009 / verification.html §4.1 Step 2 の試作実装

本 cluster は folio plugin をひとまとまりの道具として実シナリオで使い、 想定通り動くかを観察する verification.html §4.1 Step 2 (worktree integration) の試作実装。 sandbox 単体テスト (../scenarios/ + runner.sh) が hook script を mock payload で個別検証するのに対し、 本 e2e は live load 済 plugin の 4 hook + marker + SKILL を実際の Edit/Write tool 操作で発火させ、 道具としての統合挙動を end-to-end で検証する。

§1. Purpose — sandbox 単体テストとの違い

軸sandbox (../scenarios/ + runner.sh)e2e (本 cluster)
検証対象hook script 単体 (stdin payload → exit code)live load 済 plugin 全体 (4 hook + marker + SKILL)
発火方式mock JSON payload を echo | bashagent の実 Edit/Write tool 操作 (Claude Code が hook を発火)
一次 assertionexit_code + stderr_containsfile 作成有無 (PreToolUse deny → 不在 / PostToolUse notify → 存在)
実行者runner.sh (自動)agent が runbook.md を読んで手動 walk
前提yq + jqfolio plugin が load 済 session (~/.claude/plugins/folio symlink + cld auto discovery)

一次 assertion を「file 作成有無」とする理由: PreToolUse の deny は transcript に確実には現れない (research §2.6 / Issue #39344 等)。 一方 file system 上の作成/不在は決定的に観察できる。 PreToolUse deny → file 不在、 PostToolUse notify → file 存在 + 通知、 という対応で hook の発火を間接的かつ確実に判定する。

§4. 実行前提

  • folio plugin が load 済の Claude Code session であること (~/.claude/plugins/folio symlink + cld auto discovery)。 未 load の session では hook が発火せず e2e は無意味。
  • worktree 内で実行し、 probe file (architecture/spec/e2e-*.html / architecture/random-e2e/ 等) は walk 後に全削除すること (working tree を汚さない)。
  • marker file .folio/architect-active は .gitignore 済。 walk 完了後は必ず rm -f すること (残留すると sandbox の caller-marker unset→deny scenario が壊れ、 fail-open リスクも残る)。
  • sandbox 単体テスト (../scenarios/*.yaml) の全 GREEN (現 28 scenario file / 60 assertion PASS、 0 fail) を walk 前後で維持すること。
