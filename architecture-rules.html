folio architecture-rules

v0.2.0-draft · status: draft · 3-layer repo separation, directory structure, HTML markup conventions, 7-phase PR cycle

本文書は constitution.html の不変原則 (P-1〜P-12) を 実装する 規則集である。 constitution が「何が不変か」 を定義し、 本 rules が「どう構造化するか」 を規定する。

§0. Reader's Guide

spec を編集する前に FOLIO_ARCHITECT_CONTEXT=folio-architect を 必ず set (§9)。 編集後は unset で leak 防止。 新 file 追加時は §2 dir tree と §3 命名規約 を確認。 markup は §5 delta / §6 EARS / §4 front-matter 参照。

§1. 3-Layer Architecture (物理 repo 分離)

folio は 3 layer の物理 repo 分離 を採用する。 folio 自身は Layer 0 (META FRAMEWORK) のみを提供し、 Layer 1 (各 project の architecture spec) と Layer 2 (実装物) は 別 git repo として展開される。 これは ESLint flat config + Heft rig package の overlay 思想を repo 単位 に拡張したもの。

Layerrepo 位置役割変更頻度例
Layer 0 (META FRAMEWORK) folio/ repo (本 repo) spec を書くための universal rules + constitution + 7-phase + specialist agents + CI checks。 全 Layer 1/2 が consume する不変 anchor。 稀 (constitution 改訂時のみ、 MAJOR/MINOR bump) constitution.html、 architecture-rules.html、 core/agents/spec-explorer.md
Layer 1 (ARCHITECTURE SPEC) 各 project repo の architecture/ dir folio rules を適用した、 各 project の architecture spec。 project の 未来 anchor。 HTML 群 (spec/、 decisions/、 changes/、 research/、 steering/)。 project 開発中に随時 (Phase 7 G で commit) scribe repo の architecture/spec/overview.html、 TS webapp repo の architecture/spec/api-contracts.html
Layer 2 (IMPLEMENTATION) 各 project repo 全体 Layer 1 spec に従って書かれた実装物 (Claude Code plugin、 webapp、 等)。 Layer 1 は同 repo 内に同居するが、 物理 dir で分離される。 頻繁 (実装作業) scribe/ (AI 自動実装 plugin、 folio の最初の consumer)、 各 TS webapp repo

Resolution flow: Layer 2 project が folio.config.yaml で folio_version: "^1.0" を宣言し、 folio rules (Layer 0) を consume する。 Layer 1 spec は project repo 内に存在し、 folio constitution P-1〜P-12 + architecture-rules §1〜§11 を継承する。

Discovery mechanism: explicit (各 project repo root の folio.config.yaml で folio を consume すると宣言)。 convention-based 採用しない (誤検知リスク回避)。 folio repo 自身は下流 consumer を知らない = pure framework として universal を維持する (P-3 WHAT-only)。

Open-closed principle: Layer 0 (folio constitution) は P-10 (Ask first) で変更管理、 Layer 1 (各 project spec) は Layer 0 を継承 + project 固有を addition、 Layer 2 (実装) は Layer 1 spec を実装する。 上位への back-port は ADR + user 承認 MUST。

cwd で context 即判定: AI Agent は cwd が folio/ 配下なら framework 編集中、 scribe/ 配下なら scribe project 編集中、 scribe/architecture/spec/ 配下なら Layer 1 spec 編集中、 scribe/skills/ 配下なら Layer 2 実装編集中、 とすべて物理 dir で判定可能 (P-3 platform 中立 + 5 層 LLM 混同回避)。

§3. File Naming Convention

  • 全 file 名: kebab-case.html (例: spec-taxonomy.html、 universal-architect-overlay.html)。 snake_case / camelCase 禁止。
  • ADR: ADR-NNNN-<slug>.html (4 桁 ID、 ADR-0001-html-only.html)。
  • changes package: changes/<NNN>-<slug>/proposal.html (例: changes/001-constitution-amendment/proposal.html)。
  • archive: archive/changes/YYYY-MM-DD-NNN-<slug>/ で merge 後 timestamped。
  • vocabulary: stacks/<name>/vocabulary.yaml (YAML 例外、 P-2)。
  • config: folio.config.yaml (project root、 YAML 例外、 P-2)。
  • entry point: 各 cluster は README.html を持つ MUST (orphan 例外、 P-6)。

§4.3 RFC 2119 Boilerplate

全 normative section を含む file は冒頭に以下の boilerplate を持つ MUST:

<aside class="informative">
<p>本文書は IETF <strong>RFC 2119</strong> および <strong>RFC 8174 / BCP 14</strong> に定義される MUST/SHOULD/MAY/SHALL のキーワード解釈規約に従う。</p>
</aside>

§4.4 section.normative / section.informative Wrappers

MUST/SHALL を含む section は <section class="normative"> で wrap、 説明・rationale のみの section は <section class="informative"> で wrap する。 common.css が視覚的境界を表現する。

§5. Delta Marker HTML Markup

spec 改訂の trace を spec 本文に inline 保存する。 OpenSpec の ADDED/MODIFIED/REMOVED に相当する HTML-native 実装。

<!-- 追加 delta -->
<ins class="delta"
     data-delta-id="D-2026-05-21-001"
     data-delta-date="2026-05-21"
     data-delta-author="user">
  新しい要件テキスト
</ins>

<!-- 削除 delta (旧 text 保全) -->
<del class="delta"
     data-delta-id="D-2026-05-20-003"
     data-delta-superseded-by="D-2026-05-21-001">
  旧テキスト
</del>

grep pattern: grep -r 'data-delta-id' projects/ で全変更を file 横断で抽出可能。 data-delta-id は D-YYYY-MM-DD-NNN format で命名し、 同日内連番を保証する。

SemVer (constitution §3) との関係: MAJOR/MINOR bump 時は data-delta-type="major" 属性を追加し、 ADR cross-ref (data-related-adr="ADR-0042") を併記する。

§6. EARS Notation HTML Markup

acceptance criteria は EARS (Easy Approach to Requirements Syntax) の 5 pattern を採用する。

patterntemplate用途
UbiquitousThe system SHALL ...無条件不変条件
Event-drivenWHEN [trigger], the system SHALL ...event 応答
State-drivenWHILE [precondition], the system SHALL ...状態継続中
OptionalWHERE [feature included], the system SHALL ...機能オプション
UnwantedIF [unwanted condition], THEN the system SHALL ...異常応答

HTML markup:

<p class="ears" data-ears-pattern="event-driven">
  <span class="ears-id">REQ-002</span>:
  <span class="ears-when">WHEN AI agent attempts to write to spec/</span>,
  the system <span class="ears-shall">SHALL</span>
  verify FOLIO_ARCHITECT_CONTEXT environment variable.
</p>

specialist agent spec-review-ears (§8) が pattern 完全性 (5 種網羅) と ID uniqueness を CI で検証する。

§7. 7-Phase PR Cycle (folio-architect SKILL)

folio-architect は spec edit の唯一 author。 7 phase (A〜G) を順次実行する。 twill tool-architect の 7-phase pattern を universal 化したもの。

Phasenameagent並列必須
ADiscovery(なし、 folio-architect 直接)—MUST (todo list + folio.config.yaml load)
BSpec Explorationspec-explorer2-3 並列MUST
CClarifying Questions(なし、 AskUserQuestion)—MUST NOT SKIP
DSpec Designspec-architect2-3 並列optional (structural change 時のみ)
EImplementation(folio-architect 直接)—MUST (caller marker + Edit + 機械検証 + unset)
FQuality Reviewspec-review-{vocabulary,structure,ssot,temporal,ears,stakeholder}6 並列固定MUST NOT SKIP
GSummary(folio-architect 直接)—MUST (delta marker check + changelog + llms.html 更新)

並列 agent 合計: 最大 11 並列 (B 3 + D 3 + F 6 - F は除外で同時 6)、 最小 8 並列 (B 2 + F 6、 D skip 時)。

§9. Caller Marker (LLM 混同回避)

spec edit を行う agent は以下の env var を MUST set する。

# 編集前
export FOLIO_ARCHITECT_CONTEXT=folio-architect

# Edit/Write 実行 (core/spec/、 stacks/*/spec/、 projects/*/architecture/spec/ 配下)

# 編集後 (MUST、 leak 防止)
unset FOLIO_ARCHITECT_CONTEXT

core/hooks/pre-spec-write-boundary.sh が PreToolUse hook で env を検査し、 未 set / 異なる値の場合は permissionDecision: deny を返して spec edit を block する。

twill の TWL_TOOL_CONTEXT とは env var 名が異なる。 同一 host 上で twill と folio が並列存在しても hook が干渉しない (LLM 混同回避の structural guard、 P-12)。

§10. LLM 混同回避 System (3 axis × 5 層防御)

LLM (Claude Code 等) は以下の 3 つの context boundary を混同するリスクがある: (a) folio (Layer 0) vs twill (旧 plugin)、 (b) folio (Layer 0) vs 各 consumer (Layer 2 project)、 (c) Layer 1 spec edit vs Layer 2 実装 edit。 これらを 5 層防御で structural に分離する。

§10.1 5 層防御 table

層機構folio (Layer 0)scribe 等 (Layer 2)twill (旧、 reference のみ)
L1 物理分離 独立 git repo ~/projects/local-projects/folio/ (standard repo) ~/projects/local-projects/scribe/ (standard repo) ~/projects/local-projects/twill/ (bare repo、 freeze)
L2 root marker Markdown 例外 file FOLIO.md "META FRAMEWORK" SCRIBE.md (or 各 project 名) "folio consumer" (なし、 CLAUDE.md のみ)
L3 meta 宣言 HTML <meta> tag name="folio-layer" content="core" name="folio-layer" content="project" + name="folio-project" content="scribe" (なし)
L4 caller marker env var FOLIO_ARCHITECT_CONTEXT=folio-architect (folio 自身の core 編集時) FOLIO_ARCHITECT_CONTEXT=folio-architect (Layer 1 spec 編集時、 同 env var を共有) TWL_TOOL_CONTEXT (異名)
L5 directory naming folio 固有 dir core/ architecture/spec/、 architecture/howto/ 等 (Diátaxis 4 modes 物理分離) plugins/twl/、 architecture/spec/ (混在)

§10.2 Layer boundary check (consumer side)

各 Layer 2 project は folio.config.yaml を root に持ち、 caller marker hook (core/hooks/pre-spec-write-boundary.sh、 folio が提供) を install する。 hook は以下を check する:

  1. cwd が project repo 配下か (例: scribe/ 配下) — 異なるなら deny。
  2. FOLIO_ARCHITECT_CONTEXT=folio-architect が set されているか — 未 set なら deny。
  3. 編集対象 file path が architecture/spec/ 配下か (Layer 1) — 配下なら caller marker 必須、 配下でないなら check 不要。

§10.3 worktree 戦略

folio repo および各 Layer 2 project repo は standard git repo として運用する。 bare repo 構造 (twill 同型の .bare/ + main/ + worktrees/) は採用しない (twill pattern との混同回避)。 並列タスクが必要な場合は Claude Code Agent tool の isolation: "worktree" パラメータで都度生成・破棄するか、 git worktree add ~/projects/local-projects/<project>-wt/<branch> で manual に外部 dir 作成する。

§11. CI Gate Specification (將来実装)

CI workflow (.github/workflows/) が以下の gate を強制する MUST (実装は Phase 5 以降):

  1. link-integrity gate: core/checks/link-integrity.py で broken link / orphan / trace_extra / css_broken 全 0 を強制 (P-6)。
  2. declarative-form gate: core/checks/declarative-check.py で過去/未来 narration、 wave-specific note 検出時 fail (P-4)。
  3. what-only gate: core/checks/what-only-check.py で Claude Code primitives / tmux / specific path 検出時 warn or fail (P-3 / P-11)。
  4. vocabulary gate: stack vocabulary.yaml の forbidden synonym 検出 (P-5)。
  5. EARS coverage gate: spec 内 acceptance criteria に EARS 形式以外を検出時 warn (§6)。
  6. delta marker consistency gate: data-delta-id uniqueness、 ADR cross-ref 整合性 (§5)。
